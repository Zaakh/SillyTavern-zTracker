# zTracker workflow index

This document is the canonical maintainer-facing index of zTracker user workflows.

Use it when you need to answer these questions:

- Which user control triggers which handler, event, or action path?
- What state changes or visible behavior does that workflow cause?
- Which workflows already have meaningful automated coverage, and which still need stronger validation?

Keep this file focused on current behavior. Use the living specs in [specs/specs-index.md](specs/specs-index.md) for historical design intent and use [TRACKER_INJECTION_BEHAVIOR.md](TRACKER_INJECTION_BEHAVIOR.md) for the detailed runtime contract of tracker snapshot injection.

## Scope

Included:

- User-triggered settings controls in the zTracker settings drawer.
- User-triggered message, tracker, extension-menu, and character-panel controls.
- Host runtime events that have visible tracker or chat effects.

Excluded:

- Passive bootstrapping that only creates UI surfaces before the user interacts with them.
- Theme observers, internal helper composition, and other implementation details with no direct user workflow.
- The full injection-shaping matrix, which stays canonical in [TRACKER_INJECTION_BEHAVIOR.md](TRACKER_INJECTION_BEHAVIOR.md).

## Canonical sources

- [../src/components/Settings.tsx](../src/components/Settings.tsx) owns the settings surface and chat-schema metadata workflows.
- [../src/tracker.ts](../src/tracker.ts) owns rendered tracker controls and injected tracker snapshot shaping.
- [../src/ui/ui-init.ts](../src/ui/ui-init.ts) owns message-button injection, document click delegation, and host event wiring.
- [../src/ui/tracker-actions.ts](../src/ui/tracker-actions.ts) owns full generation, cancellation, edit, delete, cleanup, and the extension-menu schema popup.
- [../src/ui/character-auto-mode-exclusion.ts](../src/ui/character-auto-mode-exclusion.ts) and [../src/ui/character-panel-auto-mode.ts](../src/ui/character-panel-auto-mode.ts) own the character-panel auto-mode toggle.

## Coverage notation

- `Tests:` lists concrete automated test files that exercise the workflow or its direct runtime effect.
- `Missing` means no focused automated test was found for that workflow.
- `Tests: ...; missing dedicated UI test` means helper or runtime behavior is covered, but the exact settings-surface interaction does not appear to have its own focused test.

## Edge cases and guards to keep in mind

- Clicking the main truck button or `Regenerate Tracker` while a run is already pending for that message triggers cancellation instead of starting a second concurrent full run.
- `Skip First X Messages` is enforced for silent auto-mode flows and for first-time manual generation on untracked messages. It does not block explicit full redo of a message that already has tracker data.
- Full tracker generation resolves the current chat schema preset when one is stored in chat metadata. Partial part or field regeneration continues to use the message's saved schema preset so old messages do not silently migrate mid-edit.
- Cleanup workflows persist render-safe cleared values plus pending-target metadata instead of deleting arbitrary fields outright, because tracker rendering uses Handlebars strict mode.
- `rerenderTrackersForCurrentChat` keeps stored data when rendering fails on chat change and surfaces the problem as a message-local warning instead of silently discarding the stored tracker.
- Injection behavior in text-completion chats has terminal fallback rules that are too detailed to duplicate here. Use [TRACKER_INJECTION_BEHAVIOR.md](TRACKER_INJECTION_BEHAVIOR.md) as the runtime contract for assistant-role, virtual-character, and ambiguous terminal cases.
